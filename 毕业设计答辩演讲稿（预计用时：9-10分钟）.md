【第一部分：研究背景与需求分析】（约2分钟）
PPT：第4-5页-研究背景与痛点
首先，我想谈谈为什么要做这样一个系统。
在当今的金融市场，资讯呈现出多源、异构、碎片化的特点。传统的关键词搜索很难帮投资者做深度的逻辑分析。而近两年大语言模型（LLM）虽然理解能力强，但在金融这种对事实要求极高的垂直领域，直接使用大模型会遇到两个痛点：
第一是**“知识滞后”，大模型不知道今天刚发生的新闻； 第二是“幻觉”**，大模型在缺乏依据时可能会胡编乱造。
PPT：第6-7页-解决方案与系统特点
为了解决这两个痛点，引入 RAG（检索增强生成）技术是目前业界常用的方案。但是，目前企业级的 RAG 系统大多依赖比较复杂的分布式中间件（比如 Elasticsearch、Milvus），对于像我这样的个人开发者或者小型团队来说，部署门槛和成本有些高。
因此，我的项目目标是尝试搭建一套比较轻量化的架构。通过 Python 后端与 SQLite 单文件数据库的结合，在单机环境下跑通“新闻采集—沉淀—生成—问答”的业务流程，尽量做到易于部署和运行。
【第二部分：系统设计】（约2.5分钟）
PPT：第8页-系统框架设计
接下来汇报系统的总体设计。
系统采用了前后端分离架构。前端表现层使用了 Vue 3 搭配 Element Plus 和 Tailwind CSS，我采用了无构建工具集成的方案，可以直接通过静态页面加载，降低了本地调试的门槛。
业务逻辑层使用 FastAPI 搭建了异步 RESTful 接口，将爬虫调度、大模型生成和 RAG 检索分成了三个主要模块。
数据层则全部统一在 SQLite 中进行轻量级的持久化存储。
PPT：第9-10页-E-R图与核心表设计
在数据库设计上，除了常规的用户表和研报表，系统做了两个主要处理：
第一是新闻表（sys_news），我在原始链接（url）字段设置了 UNIQUE 唯一约束，利用数据库底层机制实现了新闻入库的去重；
第二是建立了研报快照表，即使原始新闻被清理，已经生成的研报也依然能追溯到来源。
PPT：第11页-RAG向量库与兜底策略
在 RAG 向量库的设计上，考虑到单机轻量化的需求，我没有使用独立的向量数据库，而是直接在 SQLite 中以 JSON 格式存储切片文本和向量特征。
为了保证系统运行的稳定性，这里加入了一个向量化兜底策略（Fallback）：系统在可用时优先调用 SentenceTransformer 模型提取特征；如果运行环境不支持该模型，系统会降级回退使用哈希向量（hash-fallback）。这就保证了系统在普通的 PC 桌面环境下基本都能正常运行，不会因为缺少向量模型而直接报错。
【第三部分：系统实现】（约3.5分钟）
PPT：第12页-第三部分过渡
接下来是系统的具体实现部分，我将重点汇报三个核心模块。
PPT：第13页-多源金融数据采集
首先是多源数据采集模块。面对新浪财经、第一财经等不同网站，我按站点拆分了独立的 Spider 模块，并由后端调度器根据来源分发到对应爬虫，整体上借鉴了策略解耦的思路。采集过程中结合 User-Agent 伪装，并在合适的站点使用 XPath 等方式来提取网页正文。同时，借助 FastAPI 的 BackgroundTasks，抓取任务都在后台异步执行，前端可以通过接口实时获取并展示抓取统计数据。
PPT：第14-16页-智能日报生成流
第二个模块是大模型驱动的智能日报生成。
用户在前端勾选几篇感兴趣的新闻后，后端会提取标题、时间和正文组装成上下文，并结合预设好的“金融分析师” Prompt 提示词，提交给大模型。
通过一定的格式约束，让模型输出结构化的 Markdown 格式日报。（指引屏幕 PPT 15/16页）大屏幕上展示的就是系统生成的策略分析报告，基本达到了较好的可读性。
PPT：第17-19页-RAG核心引擎：防幻觉与交互回答
第三个模块，也是这个项目中我花了最多精力学习和攻克的部分，就是 RAG 知识检索问答引擎。
系统在接收到用户提问后，会先将问题向量化，然后遍历 SQLite 里的候选切片，计算余弦相似度，召回得分最高的 Top-K 证据片段。
在将证据提供给大模型生成回答时，系统提供了“严格模式”选项：开启后，会在 Prompt 中强制规定“如果没有足够的证据支撑，就明确回答证据不足”。（指引屏幕 PPT 19页）正如演示的问答记录，这能在很大程度上减少大模型编造数据的现象。并且每次回答都附带证据链 JSON，用户可以看到回答是基于哪篇新闻得出的。

【第四部分：总结与展望】（约1分钟）
PPT：第28-29页-总结与展望
最后，对本次毕业设计进行总结与展望。
在整个项目开发过程中，我使用了 Vue3 与 FastAPI 搭建了前后端架构，编写了爬虫代码，也学习了 Prompt 工程。在最核心的 RAG 检索架构实现上，我查阅并借鉴了 GitHub 上一些优秀的开源项目和代码，结合金融资讯的具体需求，进行了修改、整合与落地。这个边学边做的过程，极大地锻炼了我的代码阅读能力和全栈工程实践能力。
当然，受限于个人能力和开发周期，系统还有许多可以改进的地方：
随着数据量的增加，SQLite 存储向量的检索性能会下降，未来有条件的话可以考虑迁移至专业的向量数据库（如 Milvus）。
目前的文本切分（Chunking）策略比较基础，对于复杂的财务表格处理得还不够好，这也是未来可以优化的方向。
未来的演进方向可以考虑引入智能体（Agent）的概念，让大模型不仅能检索新闻，还能通过工具调用获取实时的股市数据。
