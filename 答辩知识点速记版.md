# 答辩知识点速记版

## 一、系统整体怎么概括

本系统采用轻量化前后端分离架构，前端用 Vue3、Element Plus 和 Tailwind 构建页面，后端用 FastAPI 提供接口，SQLite 统一存储新闻、用户、研报和 RAG 索引数据，在单机环境下实现了“采集—沉淀—生成—问答”的完整闭环。

## 二、几个最核心的词

### 1. 无构建工具集成

前端不依赖 Vite、Webpack 之类打包工具，而是直接在 HTML 中引入 Vue3、Element Plus 和 Tailwind 的脚本文件，浏览器可以直接加载运行。

### 2. FastAPI

FastAPI 是 Python 的 Web 框架，我用它把登录、采集、日报生成和 RAG 问答等功能封装成 HTTP 接口。

### 3. 异步 RESTful 接口

异步是指接口用 `async def` 定义，更适合处理等待型任务。  
RESTful 是指接口按资源和操作来设计，例如登录、获取新闻、生成研报、删除报告等接口都有清晰的 URL 和请求方式。

### 4. BackgroundTasks

用于把耗时的爬虫任务放到后台执行，前端先拿到任务编号，再轮询任务状态，不会同步阻塞。

### 5. RAG

RAG 是“检索增强生成”，不是让模型直接凭记忆回答，而是先从本地知识库里找证据，再基于证据生成答案。

### 6. Prompt 工程

通过设计提示词来约束大模型输出，包括角色设定、格式要求、规则约束和上下文证据。

### 7. 知识滞后

大模型训练数据不是实时更新的，可能不知道当天刚发生的财经新闻。

### 8. 幻觉

大模型说得像真的，但实际上没有依据。

### 9. 严格模式

系统提供严格模式选项。开启后，如果证据不足，就明确返回“证据不足”，减少无依据扩写。

### 10. 向量化

把文本转成数字向量，方便计算相似度。

### 11. SentenceTransformer

优先使用的文本向量化模型，用来把问题和新闻切片映射到同一语义空间。

### 12. 哈希向量兜底

如果向量模型加载失败，就退回到哈希向量方案，保证系统仍然能运行。

### 13. Chunking

把长新闻切成多个小片段，便于检索和喂给模型。

### 14. 余弦相似度

比较两个向量是否接近，分数越高说明问题和文本越相关。

### 15. Top-K

只保留相似度最高的前 K 条证据，控制上下文长度。

### 16. 去重

通过数据库中 `url UNIQUE + INSERT OR IGNORE` 防止同一条新闻重复入库。

### 17. 增量存储

每次采集只补充新数据，不重复写入旧数据，也不整库重建。

### 18. 证据链 JSON

系统不仅返回答案，还返回答案所依据的新闻标题、来源、时间和片段等结构化证据信息。

19.AGENT
> **Agent（智能体）就是赋予了大模型‘手和脚’**，让它从一个只能被动阅读文本的‘大脑’，变成一个能**主动调用外部工具（比如查实时股价的API、画图工具）来完成任务的自动化助手**。”

比如说现在的 RAG 是“你喂给它新闻，它总结”；未来的 Agent 是“你让它分析茅台，它自己去调取茅台今天的K线图，自己去网上搜茅台的财报，然后整合给你看”。
## 三、老师常问的“为什么”

### 1. 为什么用 SQLite，不用 MySQL 或 Milvus

因为本课题强调轻量化、单机可部署和个人开发可实现性。SQLite 不需要单独启动数据库服务，一个文件就能存数据，部署和复现成本更低。

### 2. 为什么前后端分离

前端负责页面展示和交互，后端负责爬虫调度、数据库操作、日报生成和问答逻辑，模块边界更清晰，也更方便维护和扩展。

### 3. 为什么用 FastAPI，不用 Flask

因为 FastAPI 更适合 API 密集型项目，接口定义清晰，参数校验方便，对异步和后台任务支持也更自然。

### 4. 为什么用 Vue3

因为前端页面有很多动态状态，比如任务进度、新闻勾选、研报内容和问答消息更新，Vue3 的响应式机制更适合这种场景。

### 5. 为什么说系统是轻量化架构

因为它不用独立数据库服务，不用向量数据库，不用 Kafka、Redis、Milvus、Elasticsearch 等重型中间件，前端还能直接静态加载运行，可以在普通 PC 上部署。

6.为什么现在的“文本切分（Chunking）”太基础？复杂表格为什么处理不好？
我现在系统里用的切分方法比较简单，就是**按固定字数切分**（比如每 600 个字符切一刀，加上一点首尾重叠）。
这种方法处理纯文字的新闻还可以，但**金融研报里有大量的财务表格**。如果按固定字数硬切，很容易**把一张完整的财报表格从中间拦腰斩断**——可能表头（年份、营收、利润）被切到了上一个片段，而具体的数据被切到了下一个片段。
这样一来，当系统把切碎的片段检索出来喂给大模型时，**大模型看到的是完全错位、失去行列结构的乱码数据**，自然就无法准确回答像‘某公司第三季度净利润是多少’这样的具体问题了。
所以我说它太基础。未来更好的做法是引入**‘语义级切分（Semantic Chunking）’**，让程序能自动识别出哪里是表格、哪里是段落，**保证把一整个表格作为一个完整的片段存入数据库**，这样大模型才能看懂财务数据。

## 四、可以直接背的一段总括

本系统采用轻量化前后端分离架构，前端通过无构建工具方式集成 Vue3、Element Plus 和 Tailwind，后端使用 FastAPI 提供异步 RESTful 接口。数据层统一采用 SQLite 持久化新闻、用户、日报和 RAG 索引数据。在智能问答部分，系统通过 切分、向量化、余弦相似度和 Top-K 检索实现本地 RAG 流程，并通过 Prompt 工程和严格模式降低幻觉风险，从而在单机环境下完成“采集—沉淀—生成—问答”的完整闭环。
我的项目其实就是做了一个**‘自动化+严谨’的财经小助手**。它主要干了四件事，
**第一步是‘抓’：** 系统会去新浪财经这些网站，自动把最新的新闻爬下来。
**第二步是‘存’：** 把爬下来的新闻去重后，存到本地的数据库里。 
**第三步是‘写’：** 用户可以勾选几篇刚爬下来的新闻，让大模型根据这几篇新闻自动生成一篇总结性的市场日报。 
**第四步是‘问’：** 用户可以提问，比如‘今天原油为什么涨了？’。系统会先在本地数据库里搜出相关的新闻片段，然后把新闻和问题一起丢给大模型，强制大模型只能根据新闻来回答。这样就实现了基于本地真实资讯的问答。
