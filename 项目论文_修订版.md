第一章绪论

1.1研究背景与意义

随着互联网技术的飞速发展与全球金融市场的深度融合，金融资讯数据呈现出爆炸式增长的态势。投资者、金融分析师及相关从业人员每天需要面对海量、多源且碎片化的新闻信息。在传统的金融信息获取模式中，用户主要依赖人工浏览或基于关键词的搜索引擎来筛选信息。然而，传统关键词搜索技术存在语义理解能力差、信息召回率低以及无法进行逻辑推理等局限性，导致用户在信息检索和归纳总结上耗费大量时间，难以快速捕捉市场核心动向。

近年来，以ChatGPT、智谱GLM为代表的大语言模型（Large Language
Model,LLM）展现出了强大的自然语言理解与文本生成能力，为自动化信息摘要和智能交互提供了新的解决思路。但大语言模型在垂直领域（尤其是对数据时效性和准确性要求极高的金融领域）应用时，常面临"幻觉"（Hallucination）问题，即模型可能生成看似合理但缺乏事实依据的虚假信息，且模型权重难以实时更新以包含最新发生的新闻事件。

针对上述痛点，检索增强生成（Retrieval-Augmented
Generation,RAG）技术应运而生。RAG技术通过在外挂知识库中检索与用户提问相关的客观材料，并将其作为上下文输入给大模型，从而有效限制了模型的自由发挥，大幅提高了生成内容的准确性和时效性。因此，设计并实现一套基于大语言模型与RAG架构的轻量级金融资讯分析系统，将零散的金融新闻自动化汇聚、总结，并提供基于事实的精准问答服务，对于降低金融信息获取门槛、提升投研效率具有极其重要的现实意义与应用价值。

1.2国内外研究现状

在金融信息自动化处理领域，早期的研究主要集中于基于统计学与传统机器学习的自然语言处理（NLP）技术，如利用TF-IDF进行关键词提取，或使用TextRank算法生成新闻摘要。随着深度学习的发展，基于BERT等预训练语言模型的判别式方法在金融文本分类、情感分析中取得了显著成果，但受限于模型结构，其在开放式长文本生成与复杂问答方面表现依然欠佳。

当前，大语言模型已成为国内外人工智能研究的核心前沿。国外如OpenAI的GPT系列、Anthropic的Claude模型，国内如百度的文心一言、智谱AI的ChatGLM等，均在通用自然语言处理任务上达到了前所未有的高度。在金融垂直领域，研究人员开始探索将通用大模型应用于研报生成、财务数据分析等场景。

然而，"模型幻觉"和"知识滞后"仍是制约大模型落地的主要瓶颈。为解决这一问题，Meta（原Facebook）在2020年首次提出了RAG框架的原型，标志着基于知识检索的文本生成技术走向成熟。目前，国内外关于RAG技术的研究主要集中在向量检索优化、文档切分（Chunking）策略以及多模态检索等方向。但在实际工程落地中，许多企业级RAG系统通常依赖于庞大的分布式微服务架构（如Elasticsearch集群、Milvus向量数据库等），部署成本高昂且运维复杂。对于中小型投研团队或个人开发者而言，亟需一种低成本、易部署的轻量级RAG系统架构。本文正是在此背景下，探索一套适用于金融新闻领域的轻量化智能分析系统解决方案。

本文围绕“多源金融资讯采集—智能日报生成—RAG 问答”三段式链路，设计并实现了一个轻量级金融分析系统。系统以 Python 为核心开发语言，结合 FastAPI、SQLite 与 Vue 3 前端，实现从数据采集到知识问答的闭环。主要工作如下：

4. 轻量级前后端落地：后端提供 RESTful API，前端采用无构建链路（Vue 3 + Element Plus + Tailwind CDN），在保证功能完整的同时降低部署复杂度。

第二章核心技术与开发环境

2.1大语言模型与Prompt工程

大语言模型（LLM）是基于Transformer架构构建的深度神经网络，通过在海量互联网文本上进行无监督预训练，并结合人类反馈强化学习（RLHF）进行微调，使其具备了强大的常识储备、逻辑推理与自然语言生成能力。本项目中，智能日报生成与问答交互功能均由底层大语言模型API驱动。

由于大模型本质上是一个基于概率的文本接龙系统，其输出具有一定的不确定性。为了在金融新闻分析任务中获得稳定、高质量的输出，系统深度应用了Prompt工程（提示词工程）。Prompt工程是指通过精心设计输入文本的结构、语境和约束条件，引导大模型产生符合预期的输出。在本项目中，Prompt设计主要遵循以下原则：

角色设定（Role-playing）：为模型赋予"资深金融分析师"的身份，使其输出的语言风格更加客观、专业。

格式约束（Format
Constraint）：在生成日报时，通过Prompt强制要求模型输出Markdown或JSON格式的结构化数据，以便于前后端的解析与渲染。

上下文注入（Context
Injection）：在问答环节，通过限定词（如"请仅根据以下提供的新闻内容进行回答，如果内容中没有提及，请回答不知道"）严格限制模型的信息来源，以此作为抑制模型"幻觉"的核心手段。

2.2 RAG（检索增强生成）技术框架

RAG（Retrieval-Augmented
Generation）技术是当前解决大模型知识盲区与数据时效性问题的主流框架。其核心思想是将信息检索（Retrieval）与文本生成（Generation）相结合，赋予大模型查阅"外部参考书"的能力。

本系统中构建的RAG框架主要包含以下四个核心技术环节：

文档解析与切分（Chunking）：长篇幅的金融新闻直接输入给大模型容易超出其上下文窗口（Token限制），且会导致注意力分散。因此，需按固定长度（结合标点符号断句）及一定重叠率（Overlap）将整篇文章切分为多个内容块（Chunk）。

文本向量化（Embedding）：调用向量化模型接口，将切分后的文本块映射为高维连续向量空间中的坐标点。语义相近的文本，其在多维空间中的欧氏距离或余弦距离更近。

向量索引与相似度检索：将生成的向量及其对应的原始文本元数据（Metadata，如新闻来源、时间）存储于本地向量索引中。当系统接收到用户的问题（Query）时，使用相同的模型将Query转化为向量，并通过KNN（K近邻）或余弦相似度算法，在本地索引中快速召回Top-K个最相关的文档片段。

增强生成（Augmented
Generation）：将召回的Top-K个新闻片段作为背景上下文，与用户的原始问题组合成最终的Prompt，提交给大语言模型，生成逻辑连贯且有事实支撑的最终回答。

2.3 Web数据采集与解析技术

作为数据驱动的系统，丰富且高质量的金融新闻语料是RAG引擎发挥作用的前提。系统底层集成了定制化的Web数据采集模块（爬虫），主要依赖以下技术：

HTTP协议请求与防反爬：通过Python原生的网络请求库向目标金融站点（如新浪财经、第一财经等）发起请求。为应对基础的反爬虫机制，系统在请求头（Headers）中伪装User-Agent，并设置合理的请求间隔（Delay），以模拟正常浏览器的访问行为。

DOM树解析与数据提取：获取到网页的HTML源码后，利用正则表达式或解析库（如BeautifulSoup、XPath）遍历HTML文档的DOM（文档对象模型）树结构，精准定位并提取新闻标题、发布时间、正文内容等关键信息。

数据清洗与规范化：由于不同网站的排版格式各异，采集到的正文中往往夹杂着广告链接、冗余空格及特殊字符。系统在入库前通过文本处理脚本对这些"脏数据"进行过滤与清洗，确保后续向量化过程的文本质量。

2.4系统开发环境与前后端技术栈

本系统采用轻量级前后端分离架构。后端使用 FastAPI 提供统一接口，承担爬虫调度、日报生成、RAG 检索与问答服务；数据层使用 SQLite 本地文件数据库，实现新闻、日报、收藏与问答历史持久化。前端采用 Vue 3、Element Plus 与 TailwindCSS 的 CDN 引入方式，实现用户端与管理员端界面。

3.2.2智能摘要与日报生成需求

在信息过载的环境下，用户需要对全天候的金融事件进行快速定性与定量分析。

时间维度聚合：系统需具备基于时间窗口（如每日、每周）的数据切片筛选能力。

结构化摘要生成：必须通过预设的Prompt模板，强制大模型以多维度（如宏观经济、股市动态、行业政策）输出总结报告。

报告存档与展示：生成的日报需进行持久化存储，并在前台首页提供清晰的Markdown或HTML格式渲染展示。

3.2.3知识检索与智能问答需求

这是本系统的核心亮点，旨在解决大语言模型直接作答时产生的"幻觉"与知识滞后问题。

精准意图识别与匹配：要求系统能够准确将用户Query转化为高维向量，并利用适当的距离度量算法完成相似度检索。

防幻觉（Hallucination-free）生成：需在系统底层设置硬性约束，当知识库中检索不到相关资讯时，模型应诚实地回答"未找到相关信息"，而不得凭空捏造金融数据或虚构政策。

多轮对话上下文支持：前端界面需维护用户的历史对话记录，并在后续提问中一并作为上下文线索传递给后端，保证对话的连贯性。

数据安全性与隔离：对后台管理页面需设置基础的身份验证（鉴权）拦截，防止未经授权的用户触发敏感的爬虫任务或执行数据库删除操作。

通过对业务逻辑的解耦，本系统主要划分为三大核心模块：

Module）包含一个抽象的爬虫基类和多个针对特定站点的派生类（如news_sina、news_yicai等）。该模块通过HTTP库向目标服务器拉取DOM节点，执行正则与XPath解析，剔除广告与无用标签，最终将清洗后的字典格式数据向上层抛出。

RAG问答核心模块（RAG QA
Engine）承接文本处理的核心算法。包含：Chunking文本分割子模块、Embedding向量化子模块、Retriever检索匹配子模块，以及LLM
Generator生成子模块。此模块是保障系统问答不出现"幻觉"的技术底座。

系统管控与监控模块（Admin Control
Module）面向系统管理员，提供基于RESTful规范的后端数据接口。允许管理员通过Web前端直观地执行诸如：触发指定爬虫、查看数据库存量统计、手动重新生成当日日报等运维操作。

4.3数据库概要设计

系统数据库基于 SQLite，核心表如下：

1. sys_news：新闻主表，字段包括 id/title/content/author/time/source/scope/url/create_time，其中 url 为唯一键。
2. sys_user：用户表，字段包括 username/password/role/report_prompt/assistant_prompt 等。
3. sys_report：日报归档表，字段包括 id/user_id/title/content/create_time。
4. sys_report_source：日报-新闻来源映射表，除 report_id/news_id 外还保存来源快照字段（标题、链接、站点、时间），确保原新闻删除后仍可追溯。
5. sys_rag_doc 与 sys_rag_chunk：RAG 文档与切片向量表。
6. sys_rag_chat：问答历史表，保存用户/助手消息、模式、选中材料与证据。
4.4 RAG向量库设计策略

本系统的向量索引采用 SQLite 本地表结构而非独立向量数据库：文档级信息写入 sys_rag_doc，切片与向量写入 sys_rag_chunk。向量化流程优先使用本地 SentenceTransformer 模型（默认 BAAI/bge-small-zh-v1.5），若本地模型不可用则回退到哈希向量方案（384 维），保证系统可运行性。

文本切分采用“按段落优先拼块 + 超长硬切”策略，默认 chunk_size=680、overlap=120。检索阶段使用余弦相似度并按 top_k（可配置）返回高相关证据。
第五章系统详细设计与实现

本章是论文的核心部分，将结合上一章的架构与概要设计，详细论述基于大语言模型与RAG的金融资讯分析系统在代码层面的具体实现。本系统遵循"高内聚、轻量化"的设计原则，后端以Python为核心驱动爬虫调度与RAG推理，数据层采用SQLite本地数据库，前端采用Vue
3渐进式框架进行无构建渲染。下面将分模块阐述各部分的技术细节与核心逻辑。

5.1多源金融数据采集模块实现

数据采集模块（Web Scraper
Module）是整个系统的"数据源泉"。由于各大金融资讯网站的页面结构存在显著差异，系统在backend/spiders/目录下采用了策略模式（Strategy
Pattern），针对不同站点编写了独立的解析脚本。

5.1.1站点定制化爬虫逻辑

为了实现对多源异构数据的精准抽取，本系统针对新浪财经（news_sina.py）、第一财经（news_yicai.py）、21世纪经济报道（news_21jj.py）以及海外站点（如news_investopedia.py）等分别实现了定制化的爬虫类。

以news_sina.py为例，其核心实现逻辑如下：

网络请求封装：利用Python的requests库向目标URL发起HTTP
GET请求。为突破基础的反爬虫策略，请求头（Headers）中动态植入了常见的浏览器User-Agent标识。

DOM树解析与特征定位：获取HTML响应后，系统引入解析库（如BeautifulSoup/lxml）构建DOM树。通过分析网页结构，利用XPath或CSS选择器精准定位新闻列表的\<a\>标签获取详情页URL。

正文清洗与提取：进入新闻详情页后，提取\<title\>或特定\<h1\>标签作为新闻标题，定位包含正文的\<div\>容器。在此过程中，算法会递归剔除\<script\>、\<style\>标签及页面底部的免责声明、广告链接等冗余噪音数据，提取纯净文本。

标准化输出：将清洗后的标题、正文、发布时间及原始URL封装为统一的字典（Dict）结构，抛出给上层调度器。

5.1.2爬虫任务调度与执行引擎

后端在 main.py 中通过 POST /api/crawl 接收抓取请求，使用 FastAPI BackgroundTasks 将任务异步提交到后台执行函数。调度器根据前端传入的 source 与 scope 分发到对应爬虫模块，并统一接入状态回调，实时统计“发现/入库/跳过/失败”并通过任务事件接口返回前端。

该实现为“请求级异步后台任务”模式，并非多节点并发集群调度。任务异常由调度层捕获并写入状态，避免单个站点失败导致整体流程中断。
5.1.3 SQLite增量存储与去重实现

在 SQLite 建表 DDL 中，sys_news.url 设置了 UNIQUE 约束。数据库插入逻辑采用 INSERT OR IGNORE INTO sys_news (...)。该策略可在批量抓取时自动忽略重复 URL，且不会因唯一键冲突中断任务流程。

因此，爬虫侧可以保持“发现即尝试入库”的幂等写入方式，重复抓取会被数据库层自然拦截，前端则通过任务统计显示“入库/跳过”差异。
5.2大模型驱动的智能日报链路

5.2.1数据上下文提取与组装

日报生成由 POST /api/generate_report 触发，请求体中包含用户显式选中的 news_ids。后端根据 news_ids 批量提取新闻内容，按“标题 + 正文摘要”组装上下文文本，不采用“固定24小时自动扫描”方式。

5.2.2摘要Prompt构建与大模型交互

日报 Prompt 与模型调用位于 main.py 的 generate_report 逻辑中。系统优先读取用户配置的日报提示词（report_prompt），再与新闻上下文组合后调用大模型接口生成内容。模型返回后执行标题提取与 Markdown 标记清洗，再入库归档。

5.2.3总结持久化与前台渲染

日报写入 sys_report，并将来源新闻关系写入 sys_report_source（含来源快照）。前端通过 GET /api/reports?username=... 拉取列表，点击后调用 GET /api/report/{report_id} 获取详情与来源信息并渲染。
5.3 RAG问答核心引擎开发

5.3.1文本向量化与本地索引构建

RAG 引擎封装于 rag_engine.py，索引构建通过 build_index() 完成：先收集新闻/日报文档，再切分、向量化并写入 sys_rag_doc/sys_rag_chunk。当前默认切片参数为 chunk_size=680、overlap=120。

5.3.2用户意图向量化与相似度召回

问答接口 POST /api/rag/ask 对问题向量化后，在候选切片集合上执行余弦相似度排序，按 top_k 返回证据。系统支持 selected_only、hybrid 与 strict_selected 模式；top_k 默认 6，后端限制范围 1~12。

5.3.3上下文融合与防幻觉回答生成

问答时会将证据片段、历史会话摘要与用户问题共同组装为 Prompt。严格模式下若证据不足，将直接返回“证据不足”结论而非自由生成。当前实现为非流式一次性响应，并返回证据来源用于追溯。

7.1工作总结

在信息技术与金融行业深度融合的今天，如何从海量的资讯中快速挖掘高价值信息是业界的一大痛点。本文以解决大语言模型在垂直领域的"知识幻觉"与"时效性滞后"问题为核心出发点，设计并实现了一套基于LLM与RAG架构的金融资讯智能分析与问答系统。

在毕业设计的全生命周期中，本文主要完成了以下工作：

构建了自动化的数据采集管道：通过编写定制化爬虫策略（Spiders），实现了对多个主流财经门户的定向数据抓取，并结合SQLite的轻量级特性，完成了新闻数据的清洗与去重入盘。

落地了轻量且灵活的RAG问答引擎：在不依赖庞大微服务中间件的前提下，基于Python纯代码打通了文本分块（Chunking）、向量化（Embedding）以及相似度召回的完整链路。优化后的系统既保留了通用大模型的常识交互能力，又具备了深度结合本地新闻与智能日报进行专业、精准问答的能力。

实现了极致精简的全栈工程架构：系统后端采用Python单机架构，前端采用无构建依赖的Vue
3+Element
Plus+TailwindCSS方案。不仅大幅降低了系统的部署门槛，还保证了良好的UI交互体验和系统的易维护性。

7.2存在不足与未来展望

尽管本系统在轻量化与自动化方面达到了预期的设计目标，但受限于开发周期与单机硬件条件，系统仍存在一定的局限性，在未来的迭代中可从以下几个维度进行优化与拓展：

引入专业的向量数据库（Vector
DB）当前系统将向量特征序列化存储在本地内存或轻量级文件中，这种处理方式在数十万乃至百万级的数据规模下，将面临检索性能衰减与内存溢出的风险。未来可考虑引入专业的开源向量数据库（如Milvus、FAISS乃至Qdrant），以支持十亿级向量的亚秒级检索与持久化管理。

优化文本分块策略与多模态RAG系统目前采用了基础的按固定长度/标点符号分块策略。在面对结构极其复杂的金融研报时，粗暴的切片可能会切断表格或财务图表的语义连贯性。未来可探索"语义级Chunking"算法，并引入多模态大模型（Multi-modal
LLM），使系统具备直接解析与检索金融图表（如K线图、财务饼图）的能力。

向Agent（智能体）架构演进目前的系统逻辑较为线性（检索-\>回答）。未来可考虑将系统底座升级为Agentic（智能体）架构，赋予大模型"工具调用（Function
Calling）"的能力。例如，当用户提问某只股票的走势时，大模型不仅可以检索本地新闻，还可以自主调用外部量化API拉取实时行情数据，从而进化为一个更加全能的金融分析助手。
